# 開示汎用型マルチエージェントシステム 要件定義書

> **プロジェクト名（案）**: disclosure-multiagent / 開示変更検知アシスタント  
> **作成日**: 2026-02-23  
> **ステータス**: 要件定義（ドラフト）

---

## 1. プロジェクト概要

### 1.1 背景・課題

上場企業の経理・IR担当者は、有価証券報告書の作成にあたり以下を把握する必要がある。

- **法令・制度の改正**: 金商法、開示府令、内閣府令、ガイダンスの1年間の改正
- **新規開示項目**: 人的資本開示の拡充、SSBJ（サステナビリティ基準）、総会前開示の要請など
- **自社有報との差分**: 前年の有報に対して、どの箇所を更新・追加・削除すべきか

これらは金融庁・取引所・監査法人など複数ソースに分散しており、**情報収集と照合に多大な時間がかかる**。また、企業規模・上場市場によって求められる開示水準が異なり、**過剰開示（松すぎ）と不足開示（梅不足）のバランス**を取ることが難しい。

### 1.2 解決したいこと

**昨年の有価証券報告書を入力として、次期有報に必要な変更箇所を松竹梅で教えてくれるアプリ**を目指す。

1. **変更箇所の検出**: 1年間の金商法動向・SSBJ・関連法案全般を踏まえ、有報のどのセクションを変更すべきか、どの程度の追記・修正が必要かを示す
2. **松竹梅に応じた開示レベル**: トップ層（松）の充実した開示は理想だが、スタンダード上場など「地に足をつけた」竹・梅レベルの開示ができるよう、**松竹梅システム**で提案内容を段階化する

### 1.3 データ入力の段階的方針（セキュリティ）

| 段階 | 入力データ | 用途 | セキュリティ |
|------|------------|------|--------------|
| **初期（Phase 1〜2）** | 公開済み有報（EDINET等で既に提出・公開されたもの） | シミュレート、デモ、PoC、検証 | 公開情報のみ。クラウドLLM利用時の機密リスクが小さい |
| **最終形態** | 生データ（自社ドラフト、監査法人共有前のドラフト等） | 実務での直接利用 | ローカル処理、アクセス制御、データ保持ポリシーが必須 |

- **シミュレート方式**: 公開済み有報を入力とし、「今年版（次期有報）に必要な変更」をシミュレートする。未公開ドラフトを扱わないため、初期開発・検証時はセキュリティ負荷を抑えられる
- **最終形態**: 生データ（自社ドラフト、監査法人共有前のドラフト等）の直接利用は、ローカル処理・社内ポリシーとの整合を前提に Phase 2 以降で検討する

### 1.4 ビジョン・ゴール

> **「昨年の有報を入れると、今年の開示変更ポイントを松竹梅で教えてくれる」**

**究極のゴール**: 「結局どうすればいい？？」の実務負担から解放される究極の開示システム。**実務の進め方や記載例がその場で見られる**イメージ。専門家確認は但し書きとし、免責事項を含める。

## 2. 松竹梅システム — 開示レベルの定義

### 2.1 松竹梅の思想

| レベル | イメージ | 対象企業（目安） | 開示の特徴 |
|--------|----------|------------------|------------|
| **松** | 充実・理想 | プライム大企業、時価総額上位 | ガイダンスを上回る記載、先行開示、数値目標・KPI の詳細開示 |
| **竹** | 標準・実務 | スタンダード上場、中堅 | 必須項目を満たしつつ、過剰にならない。実務担当者が負担なく対応できる水準 |
| **梅** | 最小限・必須 | グロース、小規模上場、義務対象ギリギリ | 法令上の必須項目のみ。記載量・詳細度を最小化 |

### 2.2 松竹梅の判定軸

システムが企業の「松竹梅」をどう決めるか。

| 軸 | 松 | 竹 | 梅 |
|----|----|----|-----|
| **上場市場** | プライム | スタンダード | グロース・その他 |
| **時価総額** | 3兆円以上（SSBJ 第1段階） | 1兆円未満（目安） | 小規模（目安） |
| **開示義務の範囲** | SSBJ 義務対象・先行対応 | 人的資本必須、SSBJ 将来対象 | 人的資本必須のみ |
| **判定** | 上場市場を主軸とする。時価総額は補足。**ユーザーが「松/竹/梅」を選択**（必須）。自動推定は行わない |

### 2.3 出力の松竹梅

**同じ変更項目について、松・竹・梅で異なる提案文・記載例を出す**。

| 変更項目例 | 松の提案 | 竹の提案 | 梅の提案 |
|------------|----------|----------|----------|
| 人的資本：人材戦略 | 経営戦略との紐づけ、数値目標、KPI、第三者保証の検討 | 確保・育成・定着の方針を2〜3段落で記載。数値はあれば併記 | 「人材の確保・育成・定着に努めています」程度の最小記載 |
| 人的資本：平均給与増減率 | 算出方法の詳細、部門別・職種別の内訳、業界比較 | 数値＋算出方法の簡潔な説明（2〜3文） | 数値のみ。算出方法は「従業員の年間給与総額の平均から算出」程度 |
| SSBJ（将来対象の場合） | ギャップ分析、気候シナリオ、Scope3 の検討 | 適用時期の確認、体制・データ収集の準備状況を簡潔に | 該当しない旨の一文、または記載なし |

### 2.4 松竹梅の実装方針

- **ユーザーが選択** → 松・竹・梅のいずれかをユーザーが選択する（必須）。選択したレベルの提案をメインに、他は「より詳しく」「より簡潔に」のオプションとして表示

## 3. 機能要件

### 3.1 コア機能

| ID | 機能 | 説明 | 優先度 |
|----|------|------|--------|
| F-01 | **有報PDF入力** | 昨年度の有価証券報告書PDFを入力とする。Phase 1〜2 では**公開済み有報**に限定（ユーザーがEDINET等から取得したPDFのアップロード、またはEDINET連携での直接取得のいずれも可） | P0 |
| F-02 | **有報の構造化解析** | 入力したPDF（アップロードまたはEDINET取得）から、様式第19号等のセクション（経営方針、人的資本、サステナビリティ等）を抽出し、構造化する | P0 |
| F-03 | **法令・制度情報の取り込み** | 対象年度に基づく法令参照期間（対象年度の4/1〜翌年3/31）内の金商法改正、開示府令、人的資本ガイダンス、SSBJ関連、総会前開示など関連情報を取得・蓄積する | P0 |
| F-04 | **変更箇所の検出** | 有報の各セクションと法令・制度を照合し、「この部分を変更すべき」「この項目を追加すべき」を検出する | P0 |
| F-05 | **松竹梅別の提案** | 各変更箇所に対して、松・竹・梅の3水準の記載例・提案文を出力する | P0 |
| F-06 | **変更レポートの出力** | 検出結果をレポート形式（Markdown / PDF / HTML）で出力する | P0 |

**受け入れ基準（Phase 1）**: 定性的な基準とする。M5実証後に数値目標を追記する。例：F-02「人的資本セクションを正しく分割できる」、F-04「変更箇所を検出し根拠を明示する」、F-06「全変更箇所に根拠URLが含まれる」。

### 3.2 拡張機能（Phase 2以降）

| ID | 機能 | 説明 | 優先度 |
|----|------|------|--------|
| F-07 | **EDINET連携** | EDINETから自社または他社の有報を取得し、比較分析に使う | P1 |
| F-08 | **複数年度の比較** | 2年分・3年分の有報を比較し、トレンドや変更履歴を可視化する | P1 |
| F-09 | **差分のワード/Excel出力** | 提案文をWord・Excelテンプレートに差し込んで出力する | P1 |
| F-10 | **ユーザー設定の保存** | 松竹梅の選択、業種・上場市場などのプロファイルを保存する | P2 |
| F-11 | **監査法人・法務向けモード** | クライアント複数社の有報を一括チェックするモード | P2 |

### 3.3 対象年度と法令参照期間

- **対象年度**: ユーザーがプルダウンで選択（例：2024年度、2023年度）。直近3年程度を候補とする
- **法令参照期間**: 対象年度が2024年度の場合、2024/4/1〜2025/3/31。対象年度の開始日〜翌年3月末
- **デフォルト**: 未選択時は当年の前年度をデフォルトとする
- **決算期**: Phase 1 は **3月決算を前提**とする。12月決算等は Phase 2 で対応（下記 3.3.1 参照）

#### 3.3.1 決算期が異なる場合（Phase 2 で対応、優先度高）

| 決算期 | 法令参照期間 | 例（2024年度有報の場合） |
|--------|--------------|---------------------------|
| **3月決算** | 対象年度の 4/1 〜 翌年 3/31 | 2024/4/1〜2025/3/31 |
| **12月決算** | 対象年度の 1/1 〜 12/31 | 2024/1/1〜2024/12/31 |
| **6月決算** | 対象年度の 7/1 〜 翌年 6/30 | 2024/7/1〜2025/6/30 |
| **その他** | 決算日から1年前〜決算日 | 要個別定義 |

- **ユーザー入力**: 決算月（3/12/6 等）を選択するか、有報の表紙から自動抽出する
- **Phase 1**: 3月決算のみ。他決算期は「非対応」表示
- **Phase 2**: 12月・6月決算を追加。法令参照期間の算出ロジックを決算月に応じて分岐する

### 3.4 企業名の扱い

- **Phase 1**: レポートの企業名は**ユーザーが入力**する（オプション。未入力時は「分析対象企業」等のプレースホルダ表示）
- **Phase 2 以降**: 有報PDFの表紙から企業名を抽出することを検討する

### 3.5 情報源の範囲（法令・制度）

システムが参照する情報の範囲を定義する。

| カテゴリ | 具体的な情報 | 取得手段（案） |
|----------|-------------|----------------|
| **金商法・開示府令** | 改正法令、施行日、新規記載項目 | 金融庁HP、e-Gov、ガイダンス |
| **人的資本開示** | ガイダンス、Q&A、算定方法の例示 | 金融庁「人的資本開示に係るガイダンス」 |
| **SSBJ** | 適用スケジュール、4柱の内容、経過措置 | ASBJ、金融庁、作成要領・ハンドブック |
| **総会前開示** | 要請内容、推奨タイミング | 金融庁、取引所の開示ガイドブック |
| **取引所ルール** | プライム／スタンダード／グロースの要件差 | 各取引所の上場規程・ガイドブック |
| **EU CSRD** | 日本企業への影響パターン（子会社・域外売上） | 公式資料、解説記事（更新頻度は低め） |

### 3.6 変更種別の判定基準

| 種別 | 根拠 | 実装 |
|------|------|------|
| **追加必須** | 法令・府令で義務づけられている項目 | YAMLで種別を固定。出典種別（府令→追加必須）から自動マッピングも可 |
| **修正推奨** | ガイダンスで推奨、または改正で既存記載の見直しが求められる項目 | 同上 |
| **参考** | 任意・先行開示 | 同上 |

- Phase 1: 種別はYAMLで固定。根拠（条文・ガイダンス段落）を明示
- Phase 2: YAMLに無い項目が検出された場合、LLMに種別提案を求め「推定：要確認」として表示することを検討

## 4. マルチエージェントアーキテクチャ

### 4.1 エージェント構成案

| エージェント | 役割 | 入力 | 出力 |
|-------------|------|------|------|
| **PDF解析エージェント** | 有報PDFを読み、セクション・テーブル・テキストを構造化 | PDF ファイル | 構造化JSON（セクションID、見出し、本文、表） |
| **法令収集エージェント** | 金融庁・e-Gov等から関連法令・ガイダンスの変更を取得 | 日付範囲、キーワード | 変更事項一覧、該当URL・条文 |
| **ギャップ分析エージェント** | 有報の内容と法令要件を突き合わせ、不足・変更点を検出 | 構造化有報、法令要件 | 変更箇所リスト（セクション、変更種別、根拠） |
| **松竹梅提案エージェント** | 各変更箇所に対して、松・竹・梅の3水準の文案を生成 | 変更箇所、企業プロファイル（松竹梅選択） | 提案文（松/竹/梅別） |
| **レポート統合エージェント** | 各エージェントの出力をまとめ、レポートを生成 | 全エージェントの出力 | Markdown / HTML / PDF |

### 4.2 処理フロー

```
[ユーザー] 有報PDF + 松竹梅選択（竹）
     │
     ▼
[PDF解析エージェント]  → 構造化有報
     │
     ├──────────────────────────────────┐
     ▼                                  ▼
[法令収集エージェント]  → 法令変更一覧    [ギャップ分析エージェント]
     │                  （並列可能）              │
     └──────────────────┬───────────────────────┘
                        ▼
              [松竹梅提案エージェント]  → 提案文（竹メイン）
                        │
                        ▼
              [レポート統合エージェント]  → 最終レポート
```

### 4.3 専門家視点（ペルソナ）の組み込み

**ゴール**: 「結局どうすればいい？？」の実務負担から解放される究極の開示システム。実務の進め方や記載例がその場で見られるイメージ。専門家確認は但し書き、免責事項を含める。

専門家エージェントを独立配置せず、**既存エージェントのプロンプトに「視点」として組み込む**。企業名は使わず、汎用的な役割名とする。著作権・商標リスクを避ける。

#### 視点の定義（企業名は使わない）

| 視点 | 役割・知見 |
|------|------------|
| 公認会計士 | 会計基準、監査・開示実務の観点 |
| 取引所ルール専門家 | 上場規程、プライム/スタンダード/グロースの要件 |
| SSBJ専門家 | サステナビリティ開示基準、4柱、適用スケジュール |
| 様式・提出実務専門家 | 有報様式、提出フロー、電子開示の実務 |
| 当局・監督官庁の視点 | 金商法・府令・ガイダンス（公式発表を参照） |

#### 視点×エージェントマトリクス

| 視点 | 法令収集 | ギャップ分析 | 松竹梅提案 | レポート統合 |
|------|----------|--------------|------------|--------------|
| 公認会計士 | - | ○ | ○ | - |
| 取引所ルール専門家 | - | ○ | ○ | - |
| SSBJ専門家 | - | ○ | ○ | - |
| 様式・提出実務専門家 | PDF解析に影響 | ○ | - | ○ |
| 当局・監督官庁の視点 | ○ | ○ | - | - |

#### 著作権・利用方針

- **根拠は公式URL**: 出力に金融庁・東証・ASBJ等の公式URLを必ず記載
- **自前要約**: YAMLの内容は自らの言葉で要約。元資料の長文転載はしない
- **ペルソナは汎用表現**: プロンプト内で「公認会計士の視点で」「取引所ルールを踏まえて」等、一般的な役割として定義

#### 視点が矛盾した場合の扱い

優先順位: **当局・監督官庁 ≧ 取引所ルール ≧ 公認会計士 ≧ SSBJ専門家**。複数案が考えられる場合は併記し、ユーザーに判断を委ねる。

#### Phase別の導入

| Phase | 視点 |
|-------|------|
| Phase 1 | 公認会計士、取引所ルール、当局（最小限） |
| Phase 2 | SSBJ専門家、様式・提出実務を追加。レポートに「参照視点」を簡易表示 |
| Phase 3 | ユーザーが視点の強さを選択するオプション（検討） |

### 4.4 通信・オーケストレーション

- **イベント駆動**: 各エージェントは前段の出力をトリガーに起動。ポーリングは避ける
- **停止設計**: 法令取得失敗時は「要確認」としてスキップし、後続を継続する
- **LLM利用**: ギャップ分析・松竹梅提案はLLM（Gemini等）を利用。プロンプトに専門家視点（4.3）と「竹レベル＝過剰でない・実務的」などを明示

## 5. 非機能要件

### 5.1 セキュリティ

| 項目 | 要件 |
|------|------|
| **データの扱い（Phase 1〜2）** | 入力は**公開済み有報**に限定。公開情報のみを扱うため、クラウドLLM利用時の機密漏洩リスクが小さい |
| **データの扱い（最終形態）** | 生データ（自社ドラフト、監査法人共有前のドラフト等）を扱う場合、ローカル処理を基本とする。クラウド利用時は暗号化・アクセス制御 |
| **APIキー** | LLM・外部APIのキーは環境変数で管理。ソースコードにハードコードしない |
| **PDFの保持** | Phase 1〜2 は公開有報のため保持ポリシーは緩和可能。生データ利用時は処理後即時削除、またはユーザーが保持期間を指定 |

### 5.2 パフォーマンス

- **方針**: 処理時間の厳格な上限は設けない。**人的負担の削減**を優先する。長時間かかっても、進捗表示があれば許容する
- **進捗表示**: 必須（PDF解析→法令取得→ギャップ分析→提案生成の各ステップを表示）
- **想定範囲**: 有報100〜200ページ程度。変更箇所数に上限は設けない

### 5.3 可用性・運用

| 項目 | 要件 |
|------|------|
| **オフライン** | 法令データはキャッシュ可能。オフライン時はキャッシュのみで動作（要確認表示） |
| **ログ** | 処理の開始・終了・エラーを記録。デバッグ用 |
| **バージョン** | 法令情報の取得日時をレポートに記載し、情報の鮮度を明示する |

### 5.4 エラーハンドリング（Phase 1）

**必須で明文化する範囲**:

| カテゴリ | パターン | 対応 |
|----------|----------|------|
| **入力** | ファイル形式がPDF以外、ファイルサイズ異常、空ファイル | バリデーションで拒否、エラーメッセージ表示 |
| **処理** | PDF解析失敗、法令取得失敗、LLM呼び出し失敗 | 既出のとおり（エラー終了 or 要確認表示で継続） |
| **出力** | レポート書き込み失敗 | エラー表示、保存先の案内 |
| **環境** | APIキー未設定 | 起動時 or 初回呼び出し時に検出、エラーメッセージ |
| **UI** | 二重実行 | ボタン無効化、処理中表示 |

**推奨（設計時に考慮）**: メモリ・ディスク不足、ネットワーク断、証明書エラー、処理中のブラウザ閉じ・キャンセル。Phase 2: EDINET連携時の各パターン。

### 5.5 テスト方針（最小限）

- **Phase 1**: M5 でエンドツーエンドの動作確認。1社分の実データでレポート生成を検証する
- **単体・結合テスト**: 各エージェントの入出力を検証。詳細は開発時に定義する

## 7. データ・情報源の詳細

### 7.1 法令情報の取得方法（検討事項）

| 方法 | メリット | デメリット |
|------|----------|------------|
| **手動メンテナンス** | 確実、コスト低 | 更新が手間、遅れがち |
| **金融庁HPスクレイピング** | 自動化可能 | サイト構造変更に弱い、利用規約確認要 |
| **e-Gov API** | 公式 | 存在するか要確認 |
| **LLM + 公式URL** | 柔軟 |  hallucination リスク、引用の正確性 |
| **ハイブリッド** | 重要法令は手動、細部はLLM | 設計が複雑 |

**推奨**: Phase 1 では**手動でキーとなる改正・ガイダンスをYAML/JSONで管理**し、LLMには「この情報を基に」と渡す。Phase 2 でスクレイピングやAPI連携を検討。

### 7.2 有報の構造（様式第19号等）

解析対象の主なセクション:

- 経営方針、経営状況及び対処すべき課題等
  - 人的資本に関すること
  - サステナビリティに関すること（該当する場合）
- 事業の状況
- 設備の状況
- 提出会社の状況（従業員の状況 等）

Document AI / PyMuPDF / pdfplumber 等でテキスト抽出し、見出しパターンでセクション分割する。

### 7.3 法令情報の冗長性担保（法改正対策）

法改正が多いため、冗長性を担保する仕組みを段階的に導入する。

#### Phase 1 で導入

| 仕組み | 内容 |
|--------|------|
| **改正単位のYAML管理** | 1改正＝1エントリ。`effective_from`（施行日）, `source`（出典URL）, `change_type`（追加必須/修正推奨/参考）を持つ。後から追加しやすい設計 |
| **Gitバージョン管理** | 法令YAMLをGitで管理。変更履歴・差分を追跡し、ロールバック可能にする |
| **レポートへの日付表示** | 「法令参照: 2024/4/1〜2025/3/31（YAML取得日: YYYY-MM-DD）」をレポートに明示し、情報の鮮度を可視化 |
| **重要カテゴリの必須チェック** | 起動時、対象期間内の重要改正（人的資本・SSBJ等）がYAMLに含まれるか検証。欠けていれば警告を表示 |

#### Phase 2 で導入

| 仕組み | 内容 |
|--------|------|
| **手動YAML＋自動取得の併用** | 手動を主とし、スクレイピング/APIで補完。半自動でメンテ負荷を軽減 |
| **複数ソースの突合** | 金融庁・e-Gov等の2ソース以上で照合し、1ソースの欠落・変更に耐える |

#### Phase 3 以降・運用

改正カレンダー、定期レビュー、担当ローテーションは、運用負荷と効果を見て検討する。**持続可能性**: 自律・セルフメンテできる冗長性を確保し、法改正が続く中でも自己更新可能な仕組みを目指す。

## 8. 制約・リスク

全リスクに深刻度（高/中/低）と軽減策を記載する。

### 8.1 法務・解釈

| リスク | 深刻度 | 軽減策 |
|--------|--------|--------|
| **法令の解釈** — システムの提案をそのまま採用し、開示不備になる | 高 | レポートに「参考情報。最終判断は専門家に依頼すること」を明示。免責表示を必須化 |
| **ツール過信** — 専門家確認を怠る | 高 | レポート冒頭・UIに免責を表示。「必ず専門家確認を」を明示 |
| **利用規約・免責の不備** | 中 | 利用前に同意する形で免責を表示。必要に応じて法務レビュー |
| **LLM利用規約** — Gemini/Claude等の利用条件違反 | 低 | 各LLMの利用規約を確認。企業利用時はビジネスプラン等を検討 |

### 8.2 品質

| リスク | 深刻度 | 軽減策 |
|--------|--------|--------|
| **LLM誤引用** — 誤った法令・条文の引用 | 高 | 根拠はYAML由来のみ。URL・条文番号を必ず記載。YAMLに無い引用は出さない |
| **検出漏れ** — 必要な変更箇所を見落とす | 高 | 法令YAMLを網羅的にメンテ。複数社で検証。レポートに「要専門家確認」を明記 |
| **誤検出** — 不要な変更を提案する | 中 | 根拠を明示しユーザーが判断可能に。フィードバックでYAML改善 |
| **提案文の品質** — そのまま使えない文言 | 中 | 「ドラフト・参考」と明示。M5実証で品質確認、プロンプト改善 |

### 8.3 セキュリティ

| リスク | 深刻度 | 軽減策 |
|--------|--------|--------|
| **機密漏洩** — 生データ利用時、クラウド送信で漏洩 | 高 | Phase 1〜2は公開有報のみ。生データ時はローカルLLM or 送信範囲の制限を検討 |
| **APIキー漏洩** | 中 | 環境変数で管理。.envをgitに含めない。利用手順に注意を記載 |
| **PDFマルウェア** | 低 | 信頼ソース（EDINET等）からの利用を推奨。ウイルススキャンはOSに委譲 |

### 8.4 技術・環境

| リスク | 深刻度 | 軽減策 |
|--------|--------|--------|
| **有報の様式変更** — 様式改定で解析が崩れる | 中 | セクション分割ロジックを設定ファイル化。金融庁発表の監視で様式変更を早期把握 |
| **多様な有報** — 企業ごとの記載スタイル差で解析精度低下 | 中 | 10社以上で検証しパターン蓄積。段階的にパターン追加 |
| **LLM API提供終了・仕様変更** | 中 | 複数LLM（Gemini/Claude等）対応を検討。抽象化レイヤーで差し替え可能に |
| **企業ファイアウォール** — クラウドLLMに接続できない | 中 | Phase 1はローカル/個人利用想定。法人向けはローカルLLM or オンプレをPhase 2で検討 |
| **処理途中クラッシュ** — 不完全なレポート出力 | 中 | エラーハンドリングで中間状態をクリーンアップ。リトライ案内を表示 |

### 8.5 運用

| リスク | 深刻度 | 軽減策 |
|--------|--------|--------|
| **法令情報の更新遅れ** — YAMLの手動メンテが追いつかない | 中 | Phase 2で半自動取得を検討。重要改正はアラート運用 |

### 8.6 懸念点・疑問点（検討事項）

究極の開示システムを目指すうえで、以下の点は継続的に検討する。

| 項目 | 内容 |
|------|------|
| **「結局どうすればいい」の範囲** | 有報の変更箇所に限定するか、四半期・適時開示・総会資料まで拡張するか。Phase 1は有報に絞る |
| **ユーザーの最終アクション** | システムは「提案」を出す。Word/Excelへの反映はユーザーが行う。Phase 2のテンプレート連携で軽減を検討 |
| **専門家確認** | 但し書きとして「最終判断は専門家に依頼すること」を明示。免責事項を含める。システムは「これでOK」と断言しない |
| **持続可能性** | 自律・セルフメンテできる冗長性を確保する。法改正が続く中でも、YAML・専門家視点を自己更新可能な仕組みとする |
| **梅レベルのニーズ** | 梅＝「本当に最低限法律を守る」レベル。ニーズは必ずある |

## 9. 開発フェーズ・マイルストーン

### Phase 1: MVP（3〜4ヶ月想定）

**入力**: 公開済み有報（EDINET等から取得。シミュレート方式）

**成功基準**: 1社分の公開有報でレポートが出力できたことをもって完了とする。M5実証後に数値基準（検出件数、専門家妥当性評価等）の追記を検討する。

| マイルストーン | 内容 |
|----------------|------|
| M1 | 有報PDFのテキスト抽出・セクション分割（公開有報10社程度で検証） |
| M2 | 人的資本開示の改正要点をYAMLで定義し、ギャップ分析のプロトタイプ |
| M3 | 松竹梅提案のプロンプト設計、LLMで文案生成（竹メイン） |
| M4 | 簡易UI（Streamlit等）、レポート出力（Markdown） |
| M5 | エンドツーエンドの動作確認、1社分のレポートを実データで検証 |

### Phase 2: 拡張（3〜6ヶ月想定）

**入力**: 引き続き公開済み有報をメイン。生データ対応の検討を開始

**成功基準**: EDINETから取得した有報でレポート出力ができる。12月・6月決算に対応する。

**マイルストーン優先順位**: M6（法令半自動）→ M7（EDINET連携）→ 決算期拡張 → M9（Word連携）→ M8（プロファイル）

| マイルストーン | 内容 |
|----------------|------|
| M6 | 法令情報の半自動取得（金融庁HP等） |
| M7 | EDINET連携、公開有報の直接取得・他社比較。認証不要で取得可能な範囲。レート制限を考慮した待機・リトライ |
| M7.5 | 決算期拡張（12月・6月決算）。法令参照期間の算出を決算月に応じて分岐 |
| M8 | ユーザープロファイル（松竹梅の選択履歴等） |
| M9 | PDF / HTML エクスポート、Word テンプレート連携 |

**EDINET連携の前提**: 公表情報は認証不要で取得可能。大量取得時はレート制限・サーバー負荷に配慮する。

**生データ対応の方針**: Phase 2 では「検討」に留める。対応する場合はローカル処理を必須とし、クラウドLLM利用時は送信範囲を最小化する。対象は自社ドラフト、監査法人共有前のドラフト等。

### Phase 3: 汎用化（6ヶ月〜）

Phase 2 完了後に詳細を検討する。

| マイルストーン | 内容（概要） |
|----------------|--------------|
| M10 | 四半期報告書・適時開示資料への拡張。対象書類・法令範囲の定義が必要 |
| M11 | 監査法人・法務向けの一括チェックモード。複数社の有報を一括処理。出力形式・セキュリティの定義が必要 |
| M12 | 生データの直接利用、SaaS 化の検討。ローカル/クラウド、認証、データ保持ポリシーの定義が必要 |

## 10. 用語集

| 用語 | 説明 |
|------|------|
| **有報** | 有価証券報告書 |
| **公開済み有報** | EDINET等に既に提出・公開された有報。Phase 1〜2 の入力として使用 |
| **シミュレート** | 公開済み有報を入力とし、次期有報に必要な変更箇所・提案文を出力する方式。未公開ドラフトを扱わない |
| **生データ** | 自社の作成中ドラフト有報、監査法人共有前のドラフトなど、未公開の実データ。最終形態で直接利用を検討 |
| **金商法** | 金融商品取引法 |
| **SSBJ** | サステナビリティ・スタンダード・ベースド・ジャパニーズ（日本版サステナビリティ開示基準） |
| **松竹梅** | 本システムにおける開示レベルの3段階（松=充実、竹=標準、梅=最小限） |
| **ギャップ分析** | 現状の有報記載と法令要件を比較し、不足・変更点を特定すること |
| **対象年度** | ユーザーが選択する有報の会計年度。法令参照期間の算出に使用 |
| **法令参照期間** | 対象年度・決算期に基づく期間。3月決算は4/1〜翌年3/31。12月決算は1/1〜12/31。3.3.1 参照 |
| **決算期** | 3月/12月/6月等。Phase 1 は3月のみ。Phase 2 で12月・6月を追加 |
