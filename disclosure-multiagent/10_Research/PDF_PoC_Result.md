## 選定結論

### Phase 1 推奨: **PyMuPDF（fitz）**

**理由**:
1. **最速**: 0.717秒/社平均。100ページ超の有報でも1秒以内
2. **十分な精度**: 人的資本キーワード100%検出、文字化けゼロ
3. **シンプルな実装**: `get_text()` でページ単位のテキスト取得が直感的
4. **軽量**: 依存関係が少なくデプロイが容易

### フォールバック候補

| シナリオ | 推奨ライブラリ | 理由 |
|----------|----------------|------|
| テーブル数値（役員報酬・従業員数表）の精密抽出 | **pdfplumber** | テーブル抽出が得意 |
| LLM（Claude/Gemini）への直接渡し | **pymupdf4llm** | Markdown形式で構造が保持される |
| 処理時間が許容できる場合のギャップ分析 | **pymupdf4llm** | セクション構造の明示化が有利 |

### 実装方針（Phase 1）

```python
# メイン: PyMuPDF で全テキスト抽出
import fitz
doc = fitz.open(pdf_path)
text = "\n".join(page.get_text() for page in doc)

# 見出しパターンでセクション分割
# 例: "人的資本" "サステナビリティ" "従業員の状況" 等のキーワードで分割
import re
sections = re.split(r'(人的資本|サステナビリティ|従業員の状況)', text)
```

### 注意事項

- **company_cの事例**: 376ページの大規模有報で「人的資本」直接記載なし。全ライブラリで正しく抽出できているが、「人的資本」セクションが見つからない場合の fallback 処理が必要（「従業員の状況」等の代替キーワードに切り替え）
- **pymupdf4llm の処理時間**: company_c で98秒かかった。本番適用時はタイムアウト設定が必要

### v2.1: FUND_INDICATORS 改善（company_e 誤判定修正）

**発見した問題**: company_e はファンド運営子会社を持つ事業会社だが、有報本文に「ファンド運営子会社では...」という記述があり、旧 FUND_INDICATORS の「ファンド」単独マッチで誤検出された。

**修正内容（v2.1）**:
- 「ファンド」単独マッチを廃止
- `FUND_INDICATORS_DEFINITE`（「基準価額」「受益者」「投資信託説明書」）: 1ワード一致でファンド型確定
- `FUND_INDICATORS_STRICT`（「純資産総額」「受益証券」「分配金」等）: 3ワード以上でファンド型確定
- チェック範囲を冒頭2000→4000文字に拡大

**結果**: 10社中ファンド型 1社（company_c）のみ。誤判定0件。
