# disclosure-multiagent 実LLM出力 品質評価基準

> 作成日: 2026-02-27
> 目的: `USE_MOCK_LLM=false`（実API）実行時の出力品質を評価するための基準を定める
> 対象: M3（ギャップ分析）・M4（松竹梅提案）・M5（レポート統合）

---

## 前提: モックと実LLMの差異

| 項目 | モック（USE_MOCK_LLM=true） | 実LLM |
|------|---------------------------|-------|
| M3 ギャップ判定 | `_mock_judge_response()` による固定レスポンス | claude-haiku が法令改正内容と有報記載を照合して判定 |
| M4 提案文 | ハードコードされたサンプル文（松竹梅各約50〜300字） | claude-sonnet/haiku が実際のギャップに基づき文案生成 |
| M5 レポート | モック出力を組み合わせたレポート | 実ギャップ・実提案を元にした完全レポート |

---

## M3（ギャップ分析）評価基準

### 数量基準
- **ギャップ件数**: 5件以上（法令マスタ `laws.yaml` の `change_type: 追加必須` 件数以上が目安）
- **ギャップなし件数**: 0件以上（充足済み項目が存在する場合に記録される）

### 品質基準
| チェック項目 | 合格条件 |
|-------------|---------|
| `change_type` の正確性 | `追加必須` / `修正推奨` / `参考` の3種類のみ使用（enumバリデーション通過） |
| `gap_description` の有意性 | 「ギャップがない」等の空虚な説明ではなく、具体的な不足事項が記載される |
| 法令根拠の付与 | `law_url` フィールドに有効なURL（e-Gov等）が付与される |
| 企業固有の内容反映 | PDF（company_a.pdf等）の実際の記載内容がギャップ判定に反映される |

### モック出力との差分で確認できること
- モックはハードコード（同一入力に対し同一出力）だが、実LLMは有報内容に応じた判定がなされる
- 例: company_a.pdf の「役員報酬の説明」が実際に充足しているかを実LLMが判定

### 確認手順
```bash
cd scripts/
USE_MOCK_LLM=false python3 run_e2e.py ../10_Research/samples/company_a.pdf
# reports/ に出力されたJSONまたはMarkdownを確認
# M3出力（GapAnalysisOutput）の gaps リストを確認
```

## M4（松竹梅提案）評価基準

### 数量基準
- **提案セット数**: M3 ギャップ件数 × 3水準（松/竹/梅）の提案が生成されること
- 例: ギャップが7件なら提案セットは7個（各セットに松/竹/梅）

### 品質基準（各水準の文字数チェック）
| 水準 | 最小文字数 | 最大文字数 | 設計書定義 |
|------|-----------|-----------|-----------|
| 梅   | 50字       | 120字      | 簡潔な最低限の開示 |
| 竹   | 100字      | 260字      | 標準的な開示（100〜200字+バッファ） |
| 松   | 200字      | 480字      | 充実した詳細開示 |

### 品質確認項目
| チェック項目 | 合格条件 |
|-------------|---------|
| `level` フィールド | `松` / `竹` / `梅` の3種類のみ |
| 文字数範囲 | 上記の最小〜最大を遵守 |
| ギャップとの関連性 | 提案文がM3で特定されたギャップ（gap_description）に対応する内容 |
| 企業固有情報の反映 | 有報（PDF）の実際の記載を踏まえた文案（単なるテンプレートではない） |

### モック出力との差分で確認できること
- モック提案は特定のギャップタイプに対する固定文（例: 「当社の従業員給与は...」）
- 実LLMは M3 で特定された**実際のギャップ内容**に基づいて文案を生成する
- 文章の具体性・適切さがモックより大幅に向上するはず

## M5（レポート統合）評価基準

### 数量・形式基準
- **文字数**: 3000字以上（モックでも一定量あるが、実LLMは内容の充実度が向上）
- **構造**: 以下のセクションが含まれること
  - ギャップ分析サマリー（件数・by_change_type 内訳）
  - 個別ギャップと提案（松/竹/梅の選択が反映）
  - 注意事項（本チェックリストは参考情報等）

### 品質確認項目
| チェック項目 | 合格条件 |
|-------------|---------|
| Markdown形式 | 見出し（##/###）・表・箇条書きが適切に使用される |
| 企業名・年度 | company_a.pdf の企業情報（会社名・決算期）が正確に記載 |
| ギャップ内容の正確な引用 | M3の gap_description がレポートに正確に反映 |
| 提案水準の反映 | ユーザー選択の水準（松/竹/梅）の提案文が使用される |

## E2E実行後の確認手順（全体）

```bash
# 1. APIキー確認
python3 check_real_api.py

# 2. E2E実行（company_a.pdf）
USE_MOCK_LLM=false python3 run_e2e.py \
    ../10_Research/samples/company_a.pdf \
    --company-name "サンプル社A" \
    --fiscal-year 2023 \
    --level 竹

# 3. 出力確認
ls -la reports/
cat reports/report_*.md | wc -c   # 3000字以上を確認
grep "ギャップ\|gap" reports/report_*.md | wc -l  # ギャップ件数確認

# 4. モック実行との比較
USE_MOCK_LLM=true python3 run_e2e.py \
    ../10_Research/samples/company_a.pdf \
    --company-name "サンプル社A" \
    --fiscal-year 2023 \
    --level 竹
diff reports/report_mock_*.md reports/report_real_*.md
```

## 成功の定義（Phase 1 実LLM確認）

以下が全て満たされた場合を「実LLM動作確認 成功」とする:

1. ✅ `check_real_api.py` で API疎通確認 OK
2. ✅ M3 ギャップ件数 ≥ 5件（change_type 3種類全て使用）
3. ✅ M4 松竹梅提案の文字数が設計書範囲内
4. ✅ M5 レポートが 3000字以上かつ Markdown 形式
5. ✅ 企業固有情報（company_a.pdf の実際の記載）がレポートに反映

## 懸念事項

### CON-1: ANTHROPIC_API_KEY 未設定（現時点のブロッカー）
- 現状: WSL 環境変数に未設定
- 解除方法: 殿が `export ANTHROPIC_API_KEY=sk-ant-...` を実行
- 解除後の手順: `python3 check_real_api.py` → OK なら `run_e2e.py` 実行

### CON-2: API コスト
- M3は法令マスタ件数 × LLM呼び出し（N件）が発生
- 1回のE2E実行で haiku モデルなら数十円程度（推定）
- 事前に `USE_MOCK_LLM=true` で構造確認後に実LLM確認を推奨

### CON-3: レート制限
- company_a.pdf は712KB（4社中2番目の大きさ）
- M1 PDF解析は LLM を使わないためコスト・レート影響なし
- M3/M4 は内部で `time.sleep()` 等のレート対策が不要（haiku は高レート許容）
