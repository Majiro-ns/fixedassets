# 開示業務の業務分析（AI代替スコア付き）

> yuka-ai の 21_Clerical_Replacement_Analysis を参考に、開示汎用型マルチエージェントの対象業務を分析する。

---

## 1. 業務プロファイル

| 項目 | 内容 |
|------|------|
| **担当者** | 上場企業 経理部・IR部 担当者（スタンダード上場想定） |
| **主要業務** | 有価証券報告書の作成・更新、法令・ガイダンスに基づく開示対応 |
| **繁忙期** | 決算後〜有報提出期限（3〜4月、6月） |
| **参照先** | 金融庁HP、e-Gov、取引所ガイド、監査法人情報、自社過去有報 |

---

## 2. 1日の流れ（繁忙期）

| 時間帯 | 作業内容 | 備考 |
|--------|----------|------|
| 9:00–10:30 | 法令・ガイダンスの情報収集（HP巡回、メール確認） | 複数ソースを逐一確認 |
| 10:30–12:00 | 前年有報と改正法令の突き合わせ、変更箇所のリストアップ | 様式第19号等の該当セクションを特定 |
| 13:00–15:00 | 変更箇所ごとの記載案作成 | 人的資本・サステナビリティ等、文言の検討 |
| 15:00–16:30 | 社内レビュー対応、監査法人とのやりとり | ヒアリング・修正 |
| 16:30–18:00 | ドラフト反映、Word/PDF の体裁調整 | 手戻りが多い |

---

## 3. タスク分解と AI 代替スコア

| タスク | 作業内容 | 工数目安 | 単調度 | 判断度 | 連携度 | AI代替スコア | 代替方針 |
|--------|----------|----------|--------|--------|--------|--------------|----------|
| T1 情報収集 | 金融庁・e-Gov・取引所の改正・ガイダンスを確認 | 中 | 高 | 低 | 低 | **85** | 法令収集エージェントで取得・整理 |
| T2 有報構造把握 | 前年有報のセクション構成を把握 | 低 | 高 | 低 | 低 | **90** | PDF解析エージェントで構造化 |
| T3 変更箇所の特定 | 有報 vs 法令要件の突き合わせ | 高 | 中 | 中 | 低 | **75** | ギャップ分析エージェント |
| T4 記載案の作成 | 松竹梅に応じた文言案の作成 | 高 | 低 | 高 | 低 | **60** | 松竹梅提案エージェント（ドラフト） |
| T5 根拠の確認 | 法令・ガイダンスの該当箇所を確認 | 中 | 中 | 中 | 低 | **70** | 根拠リンク・条文番号を出力 |
| T6 社内レビュー | 他部署・監査法人との調整 | 高 | 低 | 高 | 高 | **20** | 非代替（人的業務） |
| T7 体裁調整 | Word/PDF の書式・レイアウト | 中 | 高 | 低 | 中 | **50** | Phase 2 でテンプレート連携検討 |

**スコア定義**: 単調度・判断度・連携度は 1–5。AI代替スコア = 100 - (判断度×8 + 連携度×4)、ただし単調度が高いと+15。概算。

## 4. 機能マッピング

| 既存（手作業） | 本システムで実現 | 備考 |
|----------------|------------------|------|
| HP巡回による法令確認 | 法令収集エージェント（Phase 1はYAML手動メンテ） | 後段で自動取得検討 |
| 有報の目次・該当ページの確認 | PDF解析エージェント | Document AI / PyMuPDF |
| Excel・メモでの変更箇所リスト | ギャップ分析エージェント → 変更箇所リスト | 根拠付き |
| 記載案の文言検討 | 松竹梅提案エージェント | LLM、プロンプトで竹をメインに |
| レポートの手作り | レポート統合エージェント | Markdown / HTML / PDF |
| 根拠の条文確認 | レポートにURL・条文番号を記載 | 人が最終確認 |

## 6. 工数削減の試算（目安）

| 項目 | 現状（人日/回） | 本システム利用後 | 削減率 |
|------|-----------------|------------------|--------|
| 情報収集・突き合わせ | 3–5 | 1–2（要確認・微調整） | 50–60% |
| 記載案ドラフト作成 | 2–4 | 0.5–1（レビュー前提） | 60–75% |
| 合計（有報1回分） | 5–9 | 1.5–3 | 約50–70% |

※ 最終判断・社内調整は人的作業のまま。監査法人対応は変わらず。

## 7. 市場・競合の観点

| 観点 | 内容 |
|------|------|
| **競合** | 監査法人の開示支援、有報作成支援ツール（表計算・テンプレート型）が主。AI×有報の変更検知に特化したものは少ない |
| **差別化** | 松竹梅で開示水準を段階化し、スタンダード向けの「過剰でない」提案に寄せる点 |
| **想定ユーザー** | スタンダード上場企業の経理担当者。プライム大企業は監査法人サポートが厚い |

## 8. 技術的実現性

| 要素 | 実現性 | リスク・備考 |
|------|--------|--------------|
| 有報PDFの構造化 | 高 | 様式にばらつきあり。Document AI / PyMuPDF でPoC |
| 法令の自動取得 | 中 | 金融庁HPの構造変更に弱い。Phase 1は手動YAML |
| ギャップ分析（LLM） | 中 | hallucination 対策として根拠を明示 |
| 松竹梅提案（LLM） | 中 | プロンプト設計が重要。竹＝過剰でない・実務的 |
| ローカル処理 | 高 | 機密性の観点で優先 |

---

## 9. 結論・次のアクション

1. **業務負荷のボトルネック**は T1（情報収集）と T3（変更箇所特定）。これらをAIで代替すれば工数削減効果が大きい。
2. **T4（記載案）**は LLM でドラフト作成し、人がレビュー・修正する形が現実的。
3. **Phase 1**では法令をYAMLで手動管理し、PDF解析・ギャップ分析・松竹梅提案の精度を高めることを優先する。
4. **22_MVP_Development_Checklist** に従い、PDF解析PoC → 法令YAML設計 → エージェント実装の順で進める。
