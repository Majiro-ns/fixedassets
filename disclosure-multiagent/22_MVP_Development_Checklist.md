# disclosure-multiagent MVP 開発チェックリスト

> 00_Requirements_Definition で定義された Phase 1（MVP）の実行チェックリスト。
> 3〜4ヶ月想定。エージェント単位で段階的に検証する。

## MVP 定義の確認

**MVPゴール**: 有報PDF入力（公開済み有報）→ 構造化解析 → ギャップ分析 → 松竹梅提案 → レポート出力が一連で動く

**入力**: Phase 1〜2 は公開済み有報に限定（シミュレート方式）。対象年度・松竹梅はユーザーが選択（必須）。Phase 1 は 3月決算のみ、Phase 2 で 12月・6月決算を追加

**工数削減効果（目安）**: 有報1回あたり 5–9人日 → 1.5–3人日（約50–70%削減）

**既存コードの活用率**: 0%（新規開発。multi-agent-shogun / fixed-asset-ashigaru の知見は流用検討）

## Phase 1 作業チェックリスト

### Phase 1-M1: 有報PDFのテキスト抽出・セクション分割（工数: 1–2週）

- [ ] **M1-1**: PDF解析の PoC 実施
  - Document AI vs PyMuPDF vs pdfplumber の比較検証
  - 10社程度の有報（EDINET サンプル）でテキスト抽出精度を確認
  - **ブロッカー**: サンプル有報PDFの確保（EDINETからダウンロード）

- [ ] **M1-2**: 様式第19号のセクション分割ロジック
  - 見出しパターン（経営方針、人的資本、サステナビリティ等）で分割
  - 出力形式: JSON（セクションID、見出し、本文、表）
  - 様式変更への耐性をコメントで記録

- [ ] **M1-3**: PDF解析エージェントのモジュール化
  - 入力: PDF ファイルパス
  - 出力: 構造化JSON
  - エラーハンドリング: 壊れ・画像のみ・パスワード付き→エラー終了。様式外→部分解析＋警告 or エラー（要検討）

### Phase 1-M2: 法令情報の YAML 管理（工数: 1週）

- [ ] **M2-1**: 法令情報の YAML フォーマット設計（7.3 冗長性担保に準拠）
  - 1改正＝1エントリ。フィールド: `effective_from`（施行日）, `source`（出典URL）, `change_type`（追加必須/修正推奨/参考）
  - カテゴリ: 金商法・開示府令、人的資本ガイダンス、SSBJ、総会前開示
  - Git でバージョン管理

- [ ] **M2-2**: 人的資本開示のキーとなる改正・ガイダンスを YAML 化
  - 金融庁「人的資本開示に係るガイダンス」の要点
  - Phase 1 では 5–10 件程度のエントリ

- [ ] **M2-3**: 法令収集エージェント（YAML 読み込み版）
  - 入力: 対象年度、決算期（Phase 1 は 3月固定）→ 法令参照期間 4/1〜翌年3/31、キーワード（任意）
  - 出力: 変更事項一覧、該当URL・条文
  - 起動時: 重要カテゴリ（人的資本・SSBJ等）の必須チェック。欠けていれば警告

### Phase 1-M3: ギャップ分析エージェント（工数: 2週）

- [ ] **M3-1**: ギャップ分析ロジックの設計
  - 入力: 構造化有報、法令要件リスト
  - 出力: 変更箇所リスト（セクション、変更種別、根拠、Evidence hint）
  - 変更種別: 追加必須 / 修正推奨 / 参考

- [ ] **M3-2**: LLM を用いたギャップ判定
  - プロンプト: 有報テキスト + 法令要件 → 変更の有無・根拠を出力
  - 出力形式を構造化（JSON 等）して後段で利用

- [ ] **M3-3**: 根拠の URL・条文番号の明示
  - レポートに「根拠: [ガイダンス名] (URL)」を必ず含める
  - hallucination 対策として、YAML にない引用は出さない設計

### Phase 1-M4: 松竹梅提案エージェント（工数: 2週）

- [ ] **M4-1**: 松竹梅の記載例サンプル作成
  - 人的資本: 人材戦略、平均給与増減率 等で 3 水準
  - SSBJ（将来対象）の竹・梅の扱い
  - プロンプトの few-shot 用に利用

- [ ] **M4-2**: 松竹梅提案のプロンプト設計
  - 竹＝過剰でない・実務的・スタンダード向け を明示
  - 入力: 変更箇所、企業プロファイル（松竹梅選択）
  - 出力: 提案文（松/竹/梅 各1本文）

- [ ] **M4-3**: LLM での文案生成
  - Gemini / Claude 等を利用
  - 出力の品質チェック（禁止記号、文字数目安）

### Phase 1-M5: レポート統合・UI（工数: 1–2週）

- [ ] **M5-1**: レポート統合エージェント
  - 入力: 全エージェントの出力
  - 出力: Markdown 形式のレポート（00_Requirements の 6.2 構成に準拠）

- [ ] **M5-2**: 簡易 UI（Streamlit 等）
  - アップロード画面（PDF、対象年度プルダウン、松竹梅選択・必須、企業名・任意、免責同意チェック・必須）
  - 処理進捗表示（PDF解析→法令取得→ギャップ分析→提案生成）
  - 結果表示、Markdown ダウンロード
  - レポートに「法令参照期間」「YAML取得日」を明示

- [ ] **M5-3**: エンドツーエンドの動作確認
  - 1社分の実データでレポート生成
  - 要確認・要修正の箇所を洗い出し

## エラーハンドリング（Phase 1 必須）

要件定義 5.4 に準拠。入力（ファイル形式・サイズ・空）、処理（PDF/法令/LLM）、出力（書き込み失敗）、環境（APIキー）、UI（二重実行防止）を実装する。

## サブコマンド設計（CLI 案）

```
disclosure parse <pdf_path>           # PDF解析のみ（デバッグ用）
disclosure laws list                  # 法令YAML一覧表示
disclosure analyze <pdf_path>         # ギャップ分析のみ
disclosure propose <gap_json>         # 松竹梅提案のみ（入力指定）
disclosure run <pdf_path> [--fiscal-year 2024] [--level 竹] # 一括実行（対象年度・松竹梅は必須）
disclosure serve                     # Streamlit UI 起動
```

## ブロッカー・リスク

| 項目 | 内容 | 対策 |
|------|------|------|
| 有報 PDF の多様性 | 企業により記載スタイルが異なる | 10社で検証し、パターンを蓄積 |
| LLM hallucination | 誤った法令引用 | 根拠は YAML 由来のみに限定、URL を必ず出力 |
