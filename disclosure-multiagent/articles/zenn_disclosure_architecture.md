## はじめに

「有報の人的資本開示、今年も法令が変わった。どこを直せばいい？」

上場企業の開示担当者なら、毎年この問いに向き合うことになります。内閣府令の改正、金融庁ガイダンスの更新、SSBJ（サステナビリティ基準委員会）の動向——変化のスピードに手作業での追跡が追いつかなくなってきています。

本記事では、この課題を 5つのAIエージェント（M1〜M5）のパイプラインで自動化する **disclosure-multiagent** のアーキテクチャを解説します。PDFを入力するだけで「どこにギャップがあるか」「どう書けばよいか（松・竹・梅3水準）」を出力するシステムです。

実装は [multi-agent-shogun](https://github.com/) という tmux + Claude Code の並列AI開発基盤上で進めました。215テスト全PASS、10社の有報PDFで動作確認済みです。

---

## 課題の整理

### 人的資本開示は毎年変わる

2023年度から有価証券報告書への人的資本情報の記載が義務化されました。しかし要求事項は毎年アップデートされます。

| 時期 | 主な変更点 |
|------|----------|
| 2023年3月期〜 | 人材育成方針・社内環境整備方針の記載義務化 |
| 2024年3月期〜 | 男女間賃金格差・育児休業取得率の開示義務 |
| 2025年以降 | SSBJ（サステナビリティ開示基準）の段階的適用 |

これに加えて「去年の有報にどう書いてあるか」「今年の法令と照合するとどこが足りないか」を人手で調べるのは、数百ページのPDFを読む作業です。

### LLMが得意なこと

この課題には構造的な特徴があります。

1. **入力が定型化されている**：有報PDFは「第〇部」「〇【従業員の状況】」という様式第19号に従った構成
2. **比較対象が明確**：「昨年の有報」vs「今年の法令」という差分検出タスク
3. **出力形式が決まっている**：「松・竹・梅」の改善提案文（50〜200字程度）

このような「構造化入力→LLM分析→構造化出力」はマルチエージェントのパイプラインに向いています。

## 各エージェントの詳細

### M1: PDF解析エージェント（m1_pdf_agent.py）

**責務**: 有報PDFを機械可読な構造化テキストに分解する。

有報PDFは「様式第19号」に従った定型フォーマットですが、各社の表現は様々です。M1は以下の戦略で対応します。

**キーワードによるセクション検出**

```python
JINJI_SECTION_KEYWORDS = [
    "人的資本", "人材戦略", "人材育成",      # 直接記載（優先）
    "従業員の状況", "エンゲージメント",        # 代替キーワード
    "ダイバーシティ", "女性活躍", "サステナビリティ",
]
```

13種のキーワードを優先度順に検索し、検出したセクションを `{title, page_start, text, chars}` として構造化します。

**フォントサイズによる見出し検出**

PyMuPDF の `get_text("dict")` を使い、ページ内最大フォントサイズの90%以上のスパンを見出し候補とします。「第一部」「〇【従業員の状況】」等の有報パターンとAND条件で精度を上げています。

**ファンド型有報の自動除外**

投資信託・投資法人の有報は「人的資本記載なし」が正常です。冒頭4000文字に「基準価額」「受益者」「投資信託説明書」等のファンド専用ワードが含まれる場合は自動的にフラグを立て、後続エージェントに通知します。

10社（company_a〜j）での実測：PyMuPDF 平均0.828秒/社、セクション抽出 平均27.4件/社（事業会社9社）。

> **開発の経緯**: M1は PoC実験（`pdf_poc_extract.py`）で3ライブラリ（PyMuPDF/pdfplumber/pymupdf4llm）を比較した結果を転用しています。PyMuPDFが最速（0.717秒）かつ精度も同等だったため採用。pdfplumber はテーブル抽出用フォールバックとして残してあります。

**テスト数**: 29件（境界検出・ファンド判定・各社パターン対応）

### M2: 法令収集エージェント（m2_law_agent.py）

**責務**: 法令マスタYAMLから対象期間の法令コンテキストを生成する。

**法令YAMLフォーマット**

```yaml
# law_entries_human_capital.yaml
- id: "hc-2024-001"
  category: "人的資本ガイダンス"
  change_type: "追加必須"
  effective_from: "2024-04-01"
  title: "平均給与増減率の算出方法明確化"
  source: "https://www.fsa.go.jp/..."
  summary: "連結・単体両方での開示が必要"
  target_sections: ["従業員の状況", "人的資本"]
  required_items: ["平均給与増減率", "算出根拠"]
```

1改正＝1エントリの設計により、法令変更を Git で管理できます。e-Gov API（`https://laws.e-gov.go.jp/`）との連携も設計済みで、URLからの自動収集はM6が担当します。

### M3: ギャップ分析エージェント（m3_gap_analysis_agent.py）

**責務**: M1の構造化テキストとM2の法令コンテキストを比較し、ギャップを検出する。

これが本システムの核心部分です。LLMに「この有報テキストに、この法令要件は記載されているか？」を問い、GapItemとして分類します。

**分類体系**

| 分類 | 意味 | 対応優先度 |
|------|------|---------|
| `追加必須` | 法令上必須だが記載がない | 即時対応必須 |
| `修正推奨` | 記載はあるが改善の余地あり | 次期対応推奨 |
| `参考` | 任意開示だが競合他社が記載 | 任意 |

**4層 hallucination 対策**

LLMの誤判定を防ぐため、以下の4層の制御を実装しています。

```
Layer 1: テキスト前処理
  └─ 有報テキストを 4,000文字に分割
     （先頭2,000文字 + 末尾1,000文字の重要部分を優先）

Layer 2: プロンプト設計
  └─ 「判断根拠を必ず引用文で示せ」「不明な場合は参考に分類せ」

Layer 3: 出力パース
  └─ JSON形式を強制。パース失敗時は分類を「参考」にダウングレード

Layer 4: リトライ（最大3回 + 指数バックオフ）
  └─ APIエラー・タイムアウト時の自動リトライ
```

**USE_MOCK_LLM: CI対応の要**

```bash
USE_MOCK_LLM=true python3 run_e2e.py company_a.pdf --fiscal-year 2025
```

この環境変数を設定すると、Claude APIを呼ばずにダミーのGapItemを返します。APIキーなしでパイプライン全体の結合テストができるため、CI/CDへの組み込みが容易です。

**テスト数**: 16件（各分類の境界・モック切替・リトライロジック）

### M4: 松竹梅提案エージェント（m4_proposal_agent.py）

**責務**: GapItemごとに「松・竹・梅」3水準の改善提案文を生成する。

**3水準の定義**

| 水準 | コンセプト | 文字数目安 |
|------|---------|---------|
| 松 | 業界最高水準。定量データ・中長期目標・外部評価を含む | 150〜200字 |
| 竹 | 法令要件を満たしつつ、自社の方針を具体的に説明 | 80〜150字 |
| 梅 | 法令最小要件を満たす最低限の記載 | 30〜80字 |

**few-shot設計**

提案文の質を安定させるため、各水準に実際の有報記載例（松・竹・梅それぞれ1例ずつ）をfew-shotとして渡します。

```python
FEW_SHOT_EXAMPLES = {
    "人材育成方針": {
        "松": "当社は2025年度に向けて...",  # 業界Best Practice
        "竹": "当社の人材育成方針は...",     # 標準的な記載
        "梅": "人材育成方針に基づき...",     # 最小限の記載
    }
}
```

モデルは **Claude Haiku**（`claude-haiku-4-5-20251001`）を推奨しています。提案文の生成は創造性よりも「指定フォーマットへの遵守」が重要なため、高性能モデルよりもコスト効率の良いHaikuが適しています。

**テスト数**: 41件（3水準の品質チェック・文字数・few-shot効果検証）

### M5: レポート統合エージェント（m5_report_agent.py）

**責務**: M1〜M4の出力をMarkdownレポートとして統合する。

```markdown
# 有価証券報告書 開示ギャップ分析レポート
生成日時: 2026-02-27 / 提案水準: 竹

## サマリ
- 追加必須: 3件
- 修正推奨: 2件
- 参考: 1件

## 詳細

### ① 平均給与増減率 [追加必須]
**現在の記載**: 記載なし
**法令根拠**: 人的資本ガイダンス改訂 (2024-04-01〜)

**提案文（竹水準）**:
> 前連結会計年度の従業員1人あたり平均年間給与は○○万円（前年度比+○%）であり、
> 継続的な賃上げを通じて人材の定着と採用競争力の強化を図っています。
```

**Streamlit UI**

コマンドラインに加え、`app.py` でブラウザUIも提供しています。PDFをドラッグ＆ドロップすると、M1〜M5が順次実行されて結果が表示されます。

```bash
cd scripts/
streamlit run app.py
# → http://localhost:8501
```

**テスト数**: 37件（Markdown生成・セクション順序・UI統合）

## Phase 2の拡張（M6・M7）

Phase 1（M1〜M5）の完成後、Phase 2として2つのエージェントを追加しました。

### M6: 法令URL自動収集（m6_law_url_collector.py）

金融庁・内閣府令のWebページからYAMLエントリを自動生成します。e-Gov法令API（`https://laws.e-gov.go.jp/api/1/lawdata/`）と金融庁HPのスクレイピングを組み合わせます。

### M7: EDINET連携（m7_edinet_client.py）

有報PDFをEDINETから自動取得します。開発過程で重要な発見がありました。

> **EDINET API v2** は2024年以降 `Subscription-Key` が必須になりましたが、
> **直接ダウンロードURL**（`https://disclosure2dl.edinet-fsa.go.jp/searchdocument/pdf/{DocID}.pdf`）
> は認証不要でPDFを取得できます。

この発見により、APIキーなしでも10社分の有報PDFを取得して検証を進めることができました。

### multi-agent-shogunによる並列開発

本プロジェクトは **multi-agent-shogun**（tmux + Claude Code の並列AI開発フレームワーク）で開発しました。「将軍」「家老」「足軽」の3階層で役割を分担し、M1〜M7の実装が複数の足軽エージェントにより並行して進みました。

```
将軍（プロジェクト統括）
  ├── 家老壱（足軽2/3/4を管理）
  │     ├── 足軽2: PDF解析PoC → M1実装
  │     ├── 足軽3: 法令YAML設計 → M2仕様
  │     └── 足軽4: M3/M4/M5実装
  └── 家老弐（足軽6/7/8を管理）
        └── ...
```

タスクはYAMLファイルで管理され、品質ゲート（セルフレビュー9項目 + TV-1自信度申告 + 家老スポットチェック3箇所以上）を通過したものだけがマージされます。
