## はじめに

「2027年3月期から大規模プライム企業に強制適用」——SSBJ（サステナビリティ基準委員会）開示基準のこの一文が、IR担当者・サステナビリティ担当者・経営企画部門の間で静かな緊張感を生んでいます。

従来の有価証券報告書に加え、**Scope1/2/3 GHG排出量・移行計画・シナリオ分析**といった高度な気候変動開示が義務化される。しかもこれらは定量データと定性説明が複雑に絡み合い、**担当者が手作業でチェックするには限界**があります。

本記事では、Python製マルチエージェントシステム **disclosure-multiagent** を使って、SSBJの25項目チェックを自動化する実装を解説します。`laws/ssbj_2025.yaml` に定義したチェック項目を M3ギャップ分析エージェントが有報から自動検出し、M4提案エージェントが松・竹・梅3水準の改善文案を生成します。

SSBJ担当エンジニアのゼロからの実装記録なので、法令の背景から実コードまでを包括的にカバーします。

## 1. なぜ SSBJ が重要か

### 1-1. SSBJ（サステナビリティ基準委員会）とは

SSBJ は、公益財団法人財務会計基準機構（FASF）の下に2022年7月に設立された、日本のサステナビリティ開示基準を策定する機関です。国際的には **IFRS財団・ISSB（国際サステナビリティ基準審議会）** との緊密な連携のもと、IFRS S1（全般的要求事項）・IFRS S2（気候関連開示）を日本の会計・法令体系に適合させた基準を策定しています。

2025年3月に最終確定した **SSBJ確定基準** は、日本の有価証券報告書制度（金融商品取引法）への組み込みを前提として設計されています。

### 1-2. 適用スケジュール（任意適用と強制適用）

| 対象企業 | 強制適用開始 | 提出期限 |
|---------|------------|---------|
| **大規模プライム上場企業** | **2027年3月期（2026年4月〜）** | **2028年6月** |
| その他プライム上場企業 | 2028年3月期予定 | 2029年6月予定 |
| スタンダード・グロース | 2030年3月期予定 | 2031年6月予定 |

> ⚠️ **重要な区別**: 2025年度（2025年4月〜2026年3月）は**任意適用段階**です。「早期適用が推奨される」という段階であり、2027年3月期から初めて大規模プライム企業への強制適用が始まります。

「大規模プライム」の定義（金融庁審議中）には、時価総額・純資産規模等の閾値が検討されています。東証プライム上場企業のうち資産規模上位グループが先行する見込みです。

### 1-3. TCFD からの移行：何が「強化」されるか

多くの企業がすでに **TCFD（気候関連財務情報開示タスクフォース）** 提言に基づく開示を行っています。SSBJはTCFDの4柱構造（ガバナンス/戦略/リスク管理/指標と目標）を継承しつつ、以下の点で**大幅に厳格化**されます。

| 比較項目 | TCFD提言 | SSBJ確定基準 |
|---------|---------|------------|
| 法的拘束力 | なし（推奨） | **強制適用（有報組み込み）** |
| Scope3 開示 | 任意 | 開示奨励（大規模企業は事実上必須） |
| シナリオ分析 | 推奨 | **大規模企業は必須** |
| 第三者保証 | 任意 | 開示奨励（段階的義務化検討中） |
| 移行計画 | 推奨 | **必須開示項目** |
| 財務的影響（定量） | 推奨 | **必須（定量評価を含む）** |

特に重要なのが **IFRS S2（気候関連開示）** 相当部分の義務化です。Scope1/Scope2 GHG排出量の開示は必須となり、Scope3（バリューチェーン排出量）も大規模企業には事実上求められます。

### 1-4. 人手対応の限界と AI 自動化のニーズ

SSBJ対応でAI自動化が求められる理由を整理します。

**複雑性**: SSBJのチェック項目は25件、さらにチェックリスト（CL-026〜035）で10項目が追加されます。各項目には「何を・どの程度・どのセクションに記載するか」という要件があり、有報の該当箇所を人手で漏れなくチェックするには相当の工数がかかります。

**繰り返し性**: 毎年の決算期に同じチェックを繰り返す必要があります。法令・ガイドラインの改定があれば、チェック項目も更新が必要です。

**文脈依存性**: 「移行計画を記載しているか」のような問いは、単純なキーワード検索では判定できず、文章の意味理解が必要です。

これらの課題をPythonマルチエージェントで解決するのが **disclosure-multiagent** です。

## 2. SSBJ 4柱構造の詳細解説

### 2-1. 第1柱：ガバナンス（sb-2025-001〜003）

ガバナンス開示では、「**誰が・どのように気候変動の監督を行っているか**」を明示します。

```
ガバナンス（3チェック項目）
├── sb-2025-001: 取締役会によるサステナビリティ監督体制
│   → 監督頻度・情報提供方法・意思決定プロセスへの統合
├── sb-2025-002: 経営陣の役割と責任（委員会・担当役員）
│   → サステナビリティ担当役員の職位・サステナビリティ委員会の設置状況
└── sb-2025-003: 取締役会のサステナビリティ専門知識・スキル
    → スキルマトリクスへの気候変動項目追加・研修実施状況
```

よくある開示不足パターン：
- 「サステナビリティ委員会を設置している」だけで、**取締役会への報告頻度・内容**を記載していない
- **役員報酬と気候目標の連動**（CGコードも要求）の記載なし

### 2-2. 第2柱：戦略（sb-2025-004〜010）

戦略開示は最もボリュームが大きく、7つのチェック項目があります。

```
戦略（7チェック項目）
├── sb-2025-004: 気候変動リスク・機会の特定と財務影響（定性）
│               ※移行リスク（政策・法規制・技術・市場・評判）と物理リスク（急性・慢性）の
│                 特定はsb-2025-004のrequired_itemsに含まれる
├── sb-2025-005: 気候変動リスク・機会の財務影響（定量）
├── sb-2025-006: 移行計画（短期・中期・長期）
├── sb-2025-007: シナリオ分析（1.5℃/2℃/4℃等）
├── sb-2025-008: 気候関連機会の特定と対応策（修正推奨）
├── sb-2025-009: 気候変動が事業モデル・バリューチェーンに与える影響（修正推奨）
└── sb-2025-010: サステナビリティ関連財務の統合報告との整合（参考）
```

> 📝 **表記注意**: SSBJ確定基準では「**物理リスク**」（「物理**的**リスク」ではない）という表記を使用。社内資料との統一を確認してください。

**シナリオ分析**（sb-2025-007）は、大規模プライム企業には事実上必須です。1.5℃シナリオ（パリ協定目標）と4℃シナリオ（現状趨勢）を比較し、財務影響を定量・定性で説明することが求められます。

### 2-3. 第3柱：リスク管理（sb-2025-011〜013）

```
リスク管理（3チェック項目）
├── sb-2025-011: 気候変動リスク特定・評価プロセス
├── sb-2025-012: ERM（全社リスク管理）への統合
└── sb-2025-013: 気候変動リスクの監視・レビュー体制
```

ERM（Enterprise Risk Management）への統合が重要です。「気候変動リスクは他のリスクとどう統合されているか」を説明できない企業は、この項目で不足が生じます。

### 2-4. 第4柱：指標と目標（sb-2025-014〜022）

指標と目標は最もデータ集約的な部分で、9つのチェック項目があります。

```
指標と目標（9チェック項目）
├── sb-2025-014: Scope1 GHG排出量
├── sb-2025-015: Scope2 GHG排出量（市場基準・立地基準）
├── sb-2025-016: Scope3 GHG排出量（15カテゴリ）
├── sb-2025-017: GHG削減目標と進捗（短・中・長期）
├── sb-2025-018: 気候変動対応に係る資本支出（CapEx）
├── sb-2025-019: 気候関連財務影響指標（炭素価格感応度等）
├── sb-2025-020: 内部炭素価格（ICP）の活用
├── sb-2025-021: 気候変動目標に連動した役員報酬
└── sb-2025-022: 気候変動関連の主要業績指標（KPI）
```

加えて、全般的な項目として **sb-2025-023〜025**（開示フレームワーク参照・第三者保証・SSBJ準拠宣言）があります。

## 3. disclosure-multiagent による SSBJ 自動チェック

### 3-1. 実装アーキテクチャ

disclosure-multiagent の SSBJ チェックは以下の5エージェントで構成されています。

```
laws/ssbj_2025.yaml（25件）
         │
         ▼
  M2: 法令コンテキスト取得
  適用期間・対象市場で絞り込み
         │
         ▼
  M1: PDF解析エージェント
  有報PDF → StructuredReport
  「サステナビリティに関する考え方」等のセクション抽出
         │
         ▼
  M3: ギャップ分析エージェント（Claude Haiku）
  SSBJ_KEYWORDS（25語）でSSBJ関連セクションを特定
  → 各チェック項目との照合・ギャップ判定
         │
         ▼
  M4: 松竹梅提案エージェント（Claude Haiku）
  ギャップ項目に対して3水準の改善文案生成
         │
         ▼
  M5: レポート生成
  有報担当者向けMarkdownレポート
```

### 3-2. laws/ssbj_2025.yaml の構造

25件のSSBJチェック項目はYAML形式で管理しています。

```yaml
# disclosure-multiagent — 法令・ガイダンス情報 YAML
# カテゴリ: SSBJ（サステナビリティ開示基準）
# スキーマ定義: docs/law_yaml_schema.md

version: "1.0"
effective_period:
  from: "2025-04-01"   # 任意適用期間開始
  to: "2026-03-31"

amendments:

  - id: "sb-2025-001"
    category: "SSBJ"
    change_type: "追加必須"
    effective_from: "2025-03-31"   # SSBJ確定基準発表日
    title: "SSBJ確定基準 — ガバナンス：取締役会のサステナビリティ監督体制"
    source: "https://www.ssb-j.jp/page_ssbj/pressrelease/2025/ne_20250331.html"
    summary: >
      SSBJ確定基準（2025年3月）第1柱「ガバナンス」の開示要件。
      取締役会がサステナビリティ関連リスク・機会をどのように監督しているか
      （監督頻度・情報提供方法・意思決定プロセス）を開示することが求められる。
      大規模企業は2027年3月期から強制適用。
    target_sections:
      - "サステナビリティに関する考え方及び取組"
      - "ガバナンス"
    required_items:
      - "取締役会によるサステナビリティ関連リスク・機会の監督体制の説明"
      - "取締役会への情報提供方法・頻度"
      - "取締役会の意思決定プロセスへのサステナビリティ情報の統合状況"
    applicable_markets:
      - "プライム"
    notes: "2025年度は任意適用段階。大規模企業は早期適用が推奨される"

  # ... 以下 sb-2025-002 〜 sb-2025-025 が続く
```

`change_type` の値は `追加必須` / `修正推奨` / `参考` の3種類で、ギャップ分析の優先度付けに使用します。

### 3-3. CL-026〜035: チェックリスト実装（checklist_data.json）

APIエンドポイント（`GET /api/checklist`）で返すチェックリスト項目としてSSBJの主要10件を追加しました。

| ID | 開示項目 | required | カテゴリ |
|----|---------|---------|---------|
| CL-026 | GHG排出量（Scope1・Scope2） | **必須** | サステナビリティ（SSBJ） |
| CL-027 | GHG排出量（Scope3） | 任意 | サステナビリティ（SSBJ） |
| CL-028 | 気候変動ガバナンス体制 | **必須** | サステナビリティ（SSBJ） |
| CL-029 | 気候変動リスク・機会の特定と財務影響 | **必須** | サステナビリティ（SSBJ） |
| CL-030 | 気候変動移行計画 | **必須** | サステナビリティ（SSBJ） |
| CL-031 | シナリオ分析 | 任意 | サステナビリティ（SSBJ） |
| CL-032 | GHG削減目標・進捗 | **必須** | サステナビリティ（SSBJ） |
| CL-033 | 気候関連財務影響指標 | 任意 | サステナビリティ（SSBJ） |
| CL-034 | 第三者保証（限定的/合理的保証） | 任意 | サステナビリティ（SSBJ） |
| CL-035 | SSBJ準拠宣言・適用スケジュール | 任意 | サステナビリティ（SSBJ） |

`required: true` の5件（CL-026/028/029/030/032）は必須開示項目。`required: false` の項目も大規模プライム企業には実質的に必要とされるものが多いです。

## 4. 実装コード

### 4-1. SSBJ_KEYWORDS と ALL_RELEVANCE_KEYWORDS

M3ギャップ分析エージェントは、まず有報の各セクションが「SSBJ関連かどうか」を高速にフィルタリングします。

```python
# scripts/m3_gap_analysis_agent.py より

SSBJ_KEYWORDS = [
    "SSBJ", "サステナビリティ開示", "気候変動", "気候関連",
    "GHG", "温室効果ガス", "Scope1", "Scope2", "Scope3",
    "スコープ1", "スコープ2", "スコープ3",
    "脱炭素", "カーボンニュートラル", "ネットゼロ",
    "移行計画", "移行リスク", "物理的リスク",
    "TCFD", "シナリオ分析", "排出量", "排出削減",
    "炭素", "カーボン", "再生可能エネルギー",
]  # 25キーワード

# 銀行業特化関連セクション判定キーワード（バーゼルIII / 不良債権引当）
BANKING_KEYWORDS = [
    "バーゼル", "Basel", "自己資本比率", "CET1", "Tier1", "Tier2",
    "リスク加重資産", "RWA", "LCR", "流動性カバレッジ", "NSFR",
    "不良債権", "貸倒引当金", "信用リスク", "与信",
    "要管理先", "破綻懸念先", "金利リスク", "IRRBB",
]  # 銀行・信託・証券会社向けセクション検出用

# 人的資本 + SSBJ + 銀行業キーワードを統合
ALL_RELEVANCE_KEYWORDS = HUMAN_CAPITAL_KEYWORDS + SSBJ_KEYWORDS + BANKING_KEYWORDS


def is_relevant_section(section: SectionData) -> bool:
    """
    有報セクションがSSBJ/人的資本に関連するかを判定。
    セクション見出し + 本文先頭200字でキーワードチェック。
    """
    combined = section.heading + section.text[:200]
    return any(kw in combined for kw in ALL_RELEVANCE_KEYWORDS)
```

> 📝 **設計のポイント**: `text[:200]` で先頭200字のみを見るのは意図的な設計です。セクション冒頭に主題が現れるという有報の慣習を活用し、不要なトークン消費を避けています。

`ALL_RELEVANCE_KEYWORDS` は人的資本・SSBJ・銀行業の3種類のキーワードを統合しており、金融機関の有報では `BANKING_KEYWORDS`（バーゼル・自己資本比率・不良債権引当等）によって金融機関特化セクションも検出されます。なお、実装（`m3_gap_analysis_agent.py`）では `"物理的リスク"` を使用しています。

### 4-2. ギャップ分析の実行

```python
import os
import sys
sys.path.insert(0, "scripts")

os.environ["USE_MOCK_LLM"] = "true"  # 開発時はモックモード

from m1_pdf_agent import StructuredReport, SectionData
from m2_law_agent import load_law_context
from m3_gap_analysis_agent import analyze_gaps

# 有報の「サステナビリティ」セクション例（実際はm1がPDFから抽出）
demo_section = SectionData(
    section_id="S-SSBJ-01",
    heading="サステナビリティに関する考え方及び取組",
    text=(
        "当社はカーボンニュートラルを2050年に実現することを目標としています。"
        "取締役会は年1回サステナビリティ方針を審議しています。"
        # ⚠️ Scope1/Scope2の具体的な排出量数値がない → ギャップ検出対象
    ),
    level=2,
)

demo_report = StructuredReport(
    document_id="DEMO_2025",
    company_name="サンプル株式会社",
    fiscal_year=2025,
    fiscal_month_end=3,
    sections=[demo_section],
)

# SSBJ基準を含む法令コンテキスト（laws/ssbj_2025.yaml から読み込み）
law_ctx = load_law_context(fiscal_year=2025, fiscal_month_end=3)

# ギャップ分析実行
gap_result = analyze_gaps(demo_report, law_ctx, use_mock=True)
print(f"検出ギャップ: {gap_result.summary.total_gaps} 件")

# 種別ごとの件数
for change_type, count in gap_result.summary.by_change_type.items():
    if count > 0:
        print(f"  {change_type}: {count} 件")
```

### 4-3. M4 松竹梅提案の実装（FEW_SHOT_EXAMPLES）

M4提案エージェントは、事前に定義したFew-Shot例（`FEW_SHOT_EXAMPLES`）を参照して改善文案を生成します。SSBJに対応した3セクションの例を実装しました。

**GHG排出量（Scope1・Scope2）の開示 — 松竹梅**

```python
FEW_SHOT_EXAMPLES = {
    "GHG排出量（Scope1・Scope2）の開示": {
        "松": """
【GHG排出量（2026年度実績）】

Scope1（直接排出）: [Scope1排出量]t-CO2e（前年比[率]%）
Scope2（間接排出・市場基準）: [Scope2市場基準排出量]t-CO2e
Scope2（間接排出・立地基準）: [Scope2立地基準排出量]t-CO2e
Scope1+2合計: [合計排出量]t-CO2e（前年比[率]%）

【算定方法】
GHGプロトコル「コーポレート基準」に準拠。排出係数は[係数データベース名]を使用。
組織境界は財務管理アプローチを採用し、連結子会社[N]社を含む。

【第三者検証】
[検証機関名]による限定的保証を取得（検証基準: ISO14064-3）。
""",
        "竹": (
            "Scope1（直接排出）は[Scope1排出量]t-CO2e、"
            "Scope2（市場基準）は[Scope2排出量]t-CO2eです（前年比[率]%）。"
            "GHGプロトコル「コーポレート基準」に準拠して算定しており、"
            "連結グループ全体（[N]社）を対象としています。"
        ),
        "梅": (
            "Scope1（直接排出）: [Scope1排出量]t-CO2e、"
            "Scope2（市場基準）: [Scope2排出量]t-CO2eです（GHGプロトコル準拠）。"
        ),
    },
    # ... GHG削減目標・気候変動ガバナンス も同様
}
```

**松（充実記載）**: Scope1/2の市場基準・立地基準両方、算定方法、第三者検証まで包括的に記載。
**竹（標準記載）**: 排出量数値・GHGプロトコル準拠・対象範囲を簡潔に記載。
**梅（最小記載）**: 必須項目のみ。ギャップは解消できるが、情報量は最小限。

有報担当者はギャップが検出された項目に対してこの文案をベースに、`[Scope1排出量]` 等のプレースホルダを実数値で置き換えるだけで開示文案が完成します。

## 5. Scope1/2/3 GHG排出量開示の実践

### 5-1. Scope の定義と GHGプロトコル

GHGプロトコル「コーポレート基準」が定める Scope の分類です。

| Scope | 定義 | 例 |
|-------|------|---|
| **Scope1** | 自社・グループが直接排出するGHG | 自社工場の燃料燃焼、社有車 |
| **Scope2** | 購入した電力・熱・蒸気から生じる間接排出 | 購入電力の発電由来排出 |
| **Scope3** | Scope1/2以外のバリューチェーン全体の間接排出 | 原材料調達、製品使用、廃棄 |

Scope2には**2つの算定手法**があります。
- **市場基準（Market-based）**: 購入した証書（RE100、非化石証書等）を反映
- **立地基準（Location-based）**: 地域の平均排出係数を使用

SSBJは**両方の開示を求めています**。企業が再生可能エネルギー証書を購入している場合、市場基準では大きく低下しますが、立地基準は地域の電力事情を反映した実態値になります。

### 5-2. Scope3 の15カテゴリ

Scope3はGHGプロトコルで15のカテゴリに分類されています。

```
上流 (カテゴリ1〜8)
├── Cat.1: 購入した製品・サービス
├── Cat.2: 資本財
├── Cat.3: Scope1/2に含まれない燃料・エネルギー活動
├── Cat.4: 輸送・配送（上流）
├── Cat.5: 事業から出る廃棄物
├── Cat.6: 出張
├── Cat.7: 雇用者の通勤
└── Cat.8: 上流のリースした資産

下流 (カテゴリ9〜15)
├── Cat.9: 輸送・配送（下流）
├── Cat.10: 販売した製品の加工
├── Cat.11: 販売した製品の使用
├── Cat.12: 販売した製品の廃棄
├── Cat.13: 下流のリースした資産
├── Cat.14: フランチャイズ
└── Cat.15: 投資
```

製造業ではCat.1（購入材料）とCat.11（製品使用段階）が大きく、小売業ではCat.1とCat.4が大きくなる傾向があります。SSBJは全カテゴリの開示を求めていますが、**重要性が低いカテゴリは省略理由を説明した上でスキップ可能**です。

### 5-3. 第三者保証（CL-034）の重要性

**第三者保証**（CL-034）は現時点では「任意」ですが、将来的な義務化が検討されています。2種類の保証水準があります。

| 保証水準 | 内容 | コスト感 |
|---------|------|---------|
| **限定的保証（Limited Assurance）** | 重大な誤りがないことの確認 | 比較的低コスト |
| **合理的保証（Reasonable Assurance）** | 財務監査に近い水準の詳細検証 | 高コスト |

早期適用を検討する大規模企業には、まず**限定的保証から着手**し、段階的に合理的保証に移行する戦略が現実的です。

### 5-4. 開示失敗例と M4 改善提案

**よくある失敗例**: Scope1/2の数値を記載せず、定性的な方針だけ書いてあるケース。

```
【失敗例（有報の実際のテキスト）】
「当社は気候変動をリスクと認識し、カーボンニュートラルの実現に向けて
積極的に取り組んでいます。具体的な施策については今後検討してまいります。」
```

このテキストに対して M3 がギャップを検出し、M4 が提案を生成します。

```
【M4 ギャップ検出結果】
▼ CL-026: GHG排出量（Scope1・Scope2）の開示
  判定: ギャップあり ⚠️
  根拠: テキストに「Scope1」「Scope2」の数値記載なし
  対応法令: sb-2025-014 / sb-2025-015

【M4 改善提案（竹レベル）】
Scope1（直接排出）は[Scope1排出量]t-CO2e、Scope2（市場基準）は
[Scope2排出量]t-CO2eです（前年比[率]%）。GHGプロトコル
「コーポレート基準」に準拠して算定しており、連結グループ全体（[N]社）を
対象としています。算定に使用した排出係数は[係数データベース名]に基づきます。
```

`[Scope1排出量]` 等のプレースホルダを実績値で置き換えるだけで、SSBJ基準に対応した記載文案が完成します。

## 6. 有報×EDINET連携での SSBJ 自動検証

### 6-1. D-Zenn-02（EDINET記事）との組み合わせ

本シリーズの前記事「EDINET APIで有報を自動取得 → AI法令照合パイプライン完全ガイド」で実装した EDINET 連携と、本記事の SSBJ チェックを組み合わせることで、**EDINETから有報PDFを自動取得→SSBJギャップ自動検出→改善提案生成**という完全自動パイプラインが実現します。

```python
"""
EDINET → SSBJ ギャップ分析パイプライン実行例

事前条件:
    export EDINET_SUBSCRIPTION_KEY=your-key
    export ANTHROPIC_API_KEY=sk-ant-...
"""
import sys
sys.path.insert(0, "scripts")

from m7_edinet_client import download_pdf
from m1_pdf_agent import extract_sections_from_pdf
from m2_law_agent import load_law_context
from m3_gap_analysis_agent import analyze_gaps

# Step 1: EDINET から有報 PDF を取得（認証不要）
DOC_ID = "S100VHUZ"  # トヨタ自動車 2024年度有報
pdf_path = download_pdf(DOC_ID, output_dir="downloads/pdf")

# Step 2: PDF からサステナビリティ関連セクションを抽出
report = extract_sections_from_pdf(pdf_path)
ssbj_sections = [
    s for s in report.sections
    if any(kw in s.heading + s.text[:200] for kw in ["気候変動", "GHG", "SSBJ", "サステナビリティ"])
]
print(f"SSBJ関連セクション: {len(ssbj_sections)} 件")

# Step 3: SSBJ法令コンテキスト取得
law_ctx = load_law_context(fiscal_year=2024, fiscal_month_end=3)

# Step 4: ギャップ分析（Claude Haiku で判定）
import os
os.environ["USE_MOCK_LLM"] = "false"  # 本番モード
gap_result = analyze_gaps(report, law_ctx, use_mock=False)
print(f"ギャップ件数: {gap_result.summary.total_gaps}")
print(f"対応種別: {gap_result.summary.by_change_type}")
```

### 6-2. ssbj_2025.yaml 追加後のパイプライン動作

`laws/ssbj_2025.yaml` を追加したことで、M3 ギャップ分析が以下のSSBJ項目を自動チェックするようになります。

```python

from m3_gap_analysis_agent import is_relevant_section, SectionData

# SSBJキーワードを含むセクション → 関連あり
ssbj_section = SectionData(
    section_id="test",

    text="Scope1排出量は前年比5%削減を達成しました。",
    level=2,
)

# 無関係なセクション → 関連なし

    section_id="test2",

```

SSBJ_KEYWORDS（25語）のいずれかが含まれるセクションがSSBJ関連として判定され、次の詳細なギャップ分析（Claude Haiku）に渡されます。これにより、不要なAPI呼び出しを大幅に削減できます。

## 7. まとめ：2027年3月期までのロードマップ

### 7-1. 実装のポイント 4 選

**① YAML で法令を構造化管理**
`laws/ssbj_2025.yaml` の25件を `id/category/change_type/effective_from/required_items` で管理することで、法令改定時の更新が `yaml` ファイルの編集だけで完結します。LLMプロンプトに法令テキストを直書きするより、バージョン管理・再現性が格段に向上します。

**② キーワードフィルタ → LLM の2段階設計**
`is_relevant_section()` による25語キーワードフィルタで「SSBJと無関係なセクション」を除外し、関連セクションのみを Claude Haiku に渡す設計は、**APIコストを最小化しながら精度を維持**します。全セクションをLLMに渡すと、100ページ超の有報では相当なトークンコストが発生します。

**③ 松竹梅の3水準提案で「使える文案」を生成**
「ギャップが見つかりました」で終わるツールでは担当者の作業が残ります。松（充実記載）・竹（標準記載）・梅（最小記載）の3水準でプレースホルダ入り文案を提示することで、**そのまま有報に転記できる改善案**として機能します。

**④ 物理リスク vs 物理的リスクの表記統一**
SSBJ確定基準の公式表記は「**物理リスク**」（「物理的リスク」ではない）です。社内資料・有報・AIシステムの三者で表記を統一しておかないと、キーワードマッチングに影響が出ます。現在の実装（`m3_gap_analysis_agent.py` の `SSBJ_KEYWORDS`）では `"物理的リスク"` を使用しており、SSBJ公式表記「物理リスク」との統一は今後の改善課題です。
