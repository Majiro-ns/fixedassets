# 実務上の不足点・疑問点

nencho を実務で利用する際に気になる点、未整備、要確認事項をまとめたメモ。Zenn 記事や今後の開発の優先度検討に利用する。

---

## 1. 税制・年度まわり

| 項目 | 内容 | 推奨対応 |
|------|------|----------|
| **対応年度** | `TaxYear` は現状 **R7（令和7年＝2025年）のみ**。R8 は未実装。 | 令和8年分（2026年）に向けて `TaxYear.R8` と各モジュールの分岐を追加する。税制改正の都度、表の更新が必要。 |
| **基礎控除の閾値** | `deductions.py` 内コメント: 「655万超 ≤ 2400万: 58万」について **「閾値2350万 or 2400万は要確認」** と記載あり。 | 国税庁・施行令で正式に確認し、2350万/2400万のどちらかでコードとコメントを確定する。 |
| **令和9年以降** | 基礎控除の上乗せ特例は令和7・8年分の措置。令和9年分以降は控除額が調整される旨、コメントにある。 | 法改正後にルールを追加。年度ごとの定数管理（マスタファイル or 設定）を検討してもよい。 |

## 2. セキュリティ・運用

| 項目 | 内容 | 推奨対応 |
|------|------|----------|
| **ルート .gitignore がない** | リポジトリ直下に `.gitignore` が存在しない（`web/.gitignore` のみ）。README では「nencho.db 等を .gitignore に追加」と案内している。 | ルートに `.gitignore` を追加し、`nencho.db`、`*.db-wal`、`*.db-shm`、`output/`、設定ファイル等を列挙する。 |
| **API 認証の分離** | `POST /api/calculate` は認証なしで単体計算可能。従業員一覧・一括計算は `db_path` + `password`（SecureStore）が必要。 | 設計として「計算だけ試したい」と「登録データで計算したい」の両方に対応していると明記。本番運用で API を公開する場合は、リバースプロキシでの認証や API キー等の検討をドキュメントに書く。 |
| **パスワード・DB の扱い** | パスワードはリクエスト body で送る設計。ログにパスワードや個人情報が出ないよう注意が必要。 | 既存コードでログ出力を確認し、必要ならマスクやフィルタを追加。ドキュメントに「本番では HTTPS 必須」等を追記。 |

## 3. バリデーション・品質

| 項目 | 内容 | 推奨対応 |
|------|------|----------|
| **バリデーションの範囲** | `validation_service` は基本項目・扶養・控除計算の試算まで。源泉徴収票の様式上の必須項目や、e-Tax 提出用の追加チェックは未定義。 | どのレベルまで「提出可能」とみなすかを README または capabilities に明記。必要ならチェック項目一覧を docs に追加。 |
| **R7 以外の警告** | `tax_year != 7` のときに「令和7年以外は試算値」と警告する。他年度は計算式が未実装のため結果の保証がない。 | そのまま「R7 のみ正式対応」とし、他年度は試算であることを README とスキルに明記。 |

## 4. ドキュメント・オンボーディング

| 項目 | 内容 | 推奨対応 |
|------|------|----------|
| **API スキーマの一覧** | 各ルーターの Pydantic モデルはコード内にのみ存在。OpenAPI（FastAPI の /docs）はあるが、記事や README から「どの body を送るか」を一覧で参照しづらい。 | README の「API 概要」に「リクエスト例」を 1〜2 本載せるか、`docs/api_examples.md` を追加。 |
| **CSV/Excel 取込フォーマット** | テンプレート取得 API はあるが、必須列・選択肢・サンプルファイルの説明が README にない。 | 取込用 CSV/Excel の必須列とサンプルを docs に追加。スキル「employees」からその doc を参照するよう書く。 |
| **エラーメッセージの一覧** | 計算で発生しうる `ValueError`（例: 年末調整対象外）や、バリデーションの `missing_fields` の例が一覧化されていない。 | よくあるエラーと対処を `docs/troubleshooting.md` にまとめてもよい。 |

## 6. スキル・配布まわり

| 項目 | 内容 | 推奨対応 |
|------|------|----------|
| **npx skills add の前提** | `npx skills add Majiro-ns/nencho` は、リポジトリが単体で公開され、かつ `skills/` がルート直下にあることを前提とする。 | nencho を単体リポジトリとして公開済み。README のインストール手順に実際のリポ URL を設定済み。 |
| **Claude Code プラグイン** | shinkoku には `.claude-plugin/plugin.json` がある。nencho には未整備。 | 必要に応じてプラグイン化。優先度は「スキルだけで十分か」で判断。 |

## 7. 機能・仕様の疑問点

| 項目 | 内容 | 推奨対応 |
|------|------|----------|
| **従業員の更新** | 従業員データの「更新」API が明示的でない（add のみで上書きなのか、別エンドポイントか）。 | 実装を確認し、README または API 一覧に「更新」の有無と方法を書く。 |
| **PDF/Excel の様式** | 源泉徴収票の様式が「令和7年分」に準拠している旨はコメントにある。様式番号や様式名の明記があると、税務署提出時の安心材料になる。 | `output/` または docs に「国税庁 ○○様式 令和7年分 給与所得の源泉徴収票 に準拠」等を 1 行追記。 |
| **住宅ローン控除の複数年** | 初年度・2年目以降の控除額計算が、入居年・残高に応じてどう切り替わるか。コードはあるが、利用者向けの「何年目か」の入力方法が README にない。 | 入力項目（housing_entry_year 等）と「初年度／2年目以降」の対応を README または capabilities に短く書く。 |

---

## 8. まとめ：優先して対応したいもの

1. **ルート .gitignore** — 個人情報の誤コミット防止のため即時対応推奨。
2. **基礎控除 2350万/2400万** — 税制根拠の確認とコードの確定。
3. **README のインストール先** — 単体リポ公開時にリポ URL を明記。
4. **Zenn 記事** — 免責・対象ユーザー・非対応をはっきり書く。上記の「令和7年のみ正式対応」「提出前必ず確認」を入れる。

その他は「今後の予定」「制限事項」として README や capabilities に少しずつ反映していく形でよい。
